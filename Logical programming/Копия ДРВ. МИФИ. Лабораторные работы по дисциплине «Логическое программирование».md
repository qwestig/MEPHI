# ЛАБОРАТОРНАЯ РАБОТА № 1

Генеалогические отношения и рекурсивный вывод

## **Краткое описание**

Построить базу знаний генеалогического дерева (реального или вымышленного) на языке Prolog. Реализовать предикаты для определения родственных связей произвольной степени с использованием рекурсии. Продемонстрировать работу машины вывода на нетривиальных запросах.

## **Три уровня сложности**

Уровень 1 (базовый, 20 баллов):

* БЗ минимум 15 персон в 3+ поколениях  
* Базовые предикаты: родитель/2, мужчина/1, женщина/1  
* Производные отношения: мать/2, отец/2, дедушка/2, бабушка/2, брат/2, сестра/2, дядя/2, тётя/2  
* Рекурсивные: предок/2, потомок/2  
* Сложные: двоюродный\_брат/2, троюродная\_сестра/2  
* 10+ тестовых запросов с ожидаемыми результатами

Уровень 2 (продвинутый, \+15 баллов):

* Расширение БЗ: даты рождения/смерти, места, профессии  
* Предикаты анализа:  
  * поколение/2 (номер поколения от корня дерева)  
  * общий\_предок/3 (найти ближайшего общего предка двух персон)  
  * степень\_родства/3 (расстояние в графе родства)  
* Валидация БЗ:  
  * Детектор циклов: нет\_циклов/0  
  * Проверка логичности возрастов (родитель старше ребёнка)  
  * Детектор конфликтов (персона не может быть своим предком)

Уровень 3 (экспертный, \+25 баллов):

* Автогенерация отчётов средствами Prolog:  
  * Генерация текстового описания родства: "Иван — двоюродный дедушка Марии по линии отца"  
  * Экспорт дерева в формат DOT (GraphViz) или PlantUML  
* Визуализация (один из вариантов):  
  * Интеграция с Python (pyswip) \+ matplotlib/networkx для отрисовки дерева  
  * Генерация HTML с embedded SVG через шаблоны  
  * Использование XPCE (встроенная графика SWI-Prolog)  
* Бонус: поиск всех путей между двумя персонами с выводом цепочек родства

## **Пошаговый план исполнения**

1. Проектирование БЗ: определить структуру фактов, выбрать источник данных (свои/вымышленные)  
2. Кодирование базовых предикатов: факты \+ простые правила (мать, отец, брат)  
3. Реализация рекурсии: предок/потомок с правильной обработкой базового и рекурсивного случаев  
4. Сложные отношения: двоюродные родственники через общих предков  
5. Тестирование: проверка запросов, отладка backtracking  
6. (Уровень 2\) Анализ и валидация: детектор аномалий, метрики дерева  
7. (Уровень 3\) Визуализация: выбор инструмента, реализация экспорта/отрисовки  
8. Документация: описание предикатов, примеры использования

## **Требования к оформлению и сдаче**

Состав сдаваемых материалов:

* Файл family.pl с БЗ и всеми предикатами (комментарии обязательны)  
* Файл tests.pl с запросами и ожидаемыми результатами  
* Отчёт (PDF/MD, 2-4 страницы):  
  * Описание предметной области (чьё дерево)  
  * Структура БЗ (какие факты, атрибуты)  
  * Список реализованных предикатов с сигнатурами  
  * Примеры работы (скриншоты 5-7 запросов в REPL)  
  * (Уровень 3\) Визуализация дерева (изображение)  
* Для уровня 3: код интеграции (Python-скрипт или Prolog-генератор)

Критерии оценки:

* Корректность работы предикатов (50%)  
* Полнота БЗ и тестовых случаев (20%)  
* Качество кода (читаемость, комментарии) (15%)  
* Документация (15%)  
* Бонусы за креативность: нестандартная предметная область, дополнительные метрики

Защита: устная, 10-15 минут. Вопросы на понимание рекурсии, унификации, backtracking.

# ЛАБОРАТОРНАЯ РАБОТА № 2

Составление расписания с ограничениями

## **Краткое описание**

Смоделировать расписание учебных занятий (своего потока или вымышленное) в виде БЗ Prolog. Реализовать запросы для анализа расписания. Для продвинутых уровней — применить логическое программирование с ограничениями (CLP) для автоматической генерации валидного расписания.

## **Три уровня сложности**

Уровень 1 (базовый, 20 баллов):

* БЗ: минимум 5 групп, 10 предметов, 5 преподавателей, 10 аудиторий, одна неделя  
* Факты: занятие(Группа, Предмет, Преподаватель, День, Время, Аудитория)  
* Запросы (10+ штук):  
  * Когда у группы X предмет Y?  
  * Какие аудитории свободны в среду в 10:00?  
  * У каких групп занятия ведёт преподаватель Z?  
  * Сколько пар в неделю у группы X?  
  * Какие дни свободны у преподавателя?  
* Предикаты: свободна\_аудитория/3, нагрузка\_группы/2, окна\_в\_расписании/2

Уровень 2 (продвинутый, \+15 баллов):

* Проверка корректности расписания:  
  * нет\_конфликтов\_групп/0 (группа не может быть в двух местах одновременно)  
  * нет\_конфликтов\_преподавателей/0  
  * нет\_конфликтов\_аудиторий/0 (вместимость, оборудование)  
* Оптимизационные запросы:  
  * Найти день с минимальным числом пар для группы  
  * Подсчитать окна (пустые временные слоты между парами)  
  * Найти перегруженных преподавателей (\>20 часов в неделю)  
* Предложения по улучшению: предикат можно\_переставить/2 (какие пары можно поменять местами без конфликтов)

Уровень 3 (экспертный, \+25 баллов):

* Генерация расписания с использованием CLP(FD):  
  * Входные данные: список групп, предметов, преподавателей, аудиторий, ограничений (часы по предметам, предпочтения)  
  * Ограничения:  
    * Временные слоты: 5 дней × 6 пар \= 30 слотов  
    * Никаких конфликтов (группы, преподаватели, аудитории)  
    * Равномерное распределение нагрузки  
    * Минимизация окон  
  * Выход: валидное расписание (возможно, не оптимальное, но допустимое)  
* Визуализация: экспорт в HTML-таблицу или CSV для Excel  
* Бонус: оптимизация по критерию (минимум окон, максимум компактности дня)

## **Пошаговый план исполнения**

1. Сбор данных: реальное расписание потока или синтетические данные  
2. Моделирование: проектирование структуры фактов (что хранить, какие атрибуты)  
3. Кодирование БЗ: ввод фактов в Prolog  
4. Базовые запросы: реализация предикатов для анализа  
5. (Уровень 2\) Валидация: предикаты детекции конфликтов, метрики качества  
6. (Уровень 3\) CLP: изучение библиотеки clpfd, формализация ограничений  
7. (Уровень 3\) Генерация и визуализация: запуск решателя, экспорт результатов  
8. Документация: описание модели, примеры запросов

## **Требования к оформлению и сдаче**

Состав материалов:

* schedule.pl — БЗ и предикаты  
* queries.pl — тестовые запросы (с комментариями, что проверяют)  
* Отчёт (PDF/MD, 3-5 страниц):  
  * Описание предметной области (чьё расписание, характеристики)  
  * Схема БЗ (структура фактов, домены значений)  
  * Реализованные запросы (примеры работы)  
  * (Уровень 2\) Результаты валидации (найденные конфликты/рекомендации)  
  * (Уровень 3\) Описание ограничений CLP, скриншот сгенерированного расписания  
* Для уровня 3: сгенерированное расписание (таблица HTML/CSV)

Критерии оценки:

* Корректность модели и запросов (45%)  
* Полнота БЗ (реалистичность данных) (20%)  
* Качество кода (15%)  
* Документация (15%)  
* (Уровень 3\) Работоспособность CLP-решения (5% бонус)

Защита: демонстрация запросов в REPL, объяснение работы CLP (если применялось).

# ЛАБОРАТОРНАЯ РАБОТА № 3

Моделирование знаний с НЕ-факторами

## **Краткое описание**

Разработать экспертную систему для предметной области с неполнотой, неопределённостью или противоречивостью. Реализовать полный цикл инженерии знаний: приобретение (из экспертных правил или данных), представление (с использованием вероятностного ЛП, ASP или нечёткой логики), обработку (вывод с объяснениями), валидацию.

## **Три уровня сложности**

Уровень 1 (базовый, 20 баллов):

* Выбор одной предметной области:  
  * Медицинская диагностика (симптомы → болезни)  
  * Техническая диагностика (показания датчиков → неисправности автомобиля/робота)  
  * Рекомендательная система (профиль пользователя → рекомендации)  
* Минимум 10 сущностей (симптомов/датчиков), 5 классов (диагнозов/неисправностей)  
* Представление неопределённости одним методом:  
  * ProbLog: вероятностные факты 0.8::симптом(температура)  
  * NAF \+ default logic: правила с отрицанием \\+  
  * Нечёткая логика: функции принадлежности температура(высокая, 0.7)  
* Реализация вывода: предикат диагноз/2 с объяснением (почему этот диагноз)  
* Тестирование на 5+ кейсах с разбором результатов

Уровень 2 (продвинутый, \+15 баллов):

* Комбинирование нескольких подходов:  
  * ProbLog для вероятностей \+ NAF для закрытого мира  
  * ASP для разрешения противоречий \+ вероятности  
* Приобретение знаний:  
  * Экспертные правила: формализация текстовых описаний в Prolog  
  * Или: извлечение паттернов из синтетического датасета (if-then правила)  
* Объяснения:  
  * Трассировка вывода: показать цепочку правил, приведших к заключению  
  * Confidence/вероятность финального вывода  
* Валидация БЗ:  
  * Проверка полноты (все ли комбинации симптомов покрыты)  
  * Проверка непротиворечивости (нет взаимоисключающих диагнозов с высокой вероятностью)

Уровень 3 (экспертный, \+25 баллов):

* Реализация ReAct-цикла для автономного агента:  
  * Perceive: симуляция показаний датчиков (генерация с шумом/пропусками)  
  * Reason: применение правил с НЕ-факторами для принятия решения  
  * Act: вывод команды (логирование)  
  * Цикл повторяется N итераций с изменением состояния  
* Сравнение подходов:  
  * Реализовать ту же задачу на ProbLog И на ASP  
  * Сравнить результаты (точность, объяснимость, время работы)  
  * Таблица: когда какой подход эффективнее  
* Визуализация:  
  * Граф вывода (какие правила сработали, с какими вероятностями)  
  * Дерево решений для диагностики  
* Бонус: интеграция с Python для интерактивного интерфейса (ввод симптомов → вывод диагноза в реальном времени)

## **Пошаговый план исполнения**

1. Выбор предметной области: исследование задачи, определение сущностей и отношений  
2. Приобретение знаний: сбор экспертных правил или генерация синтетических данных  
3. Формализация: перевод правил в Prolog/ProbLog/ASP синтаксис  
4. Реализация вывода: базовые предикаты для рассуждений  
5. (Уровень 2\) Трассировка и объяснения: механизм логирования цепочек вывода  
6. (Уровень 2\) Валидация: предикаты проверки БЗ на полноту/непротиворечивость  
7. (Уровень 3\) ReAct-цикл или сравнение подходов: реализация, эксперименты  
8. Тестирование и анализ: прогон кейсов, оценка качества работы системы  
9. Документация: описание системы, результаты экспериментов

## **Требования к оформлению и сдаче**

Состав материалов:

* knowledge\_base.pl (или .problog, .lp для ASP) — БЗ с правилами  
* inference.pl — предикаты вывода и объяснений  
* tests.pl — тестовые кейсы (минимум 5 сценариев)  
* Отчёт (PDF/MD, 5-8 страниц):  
  * Описание предметной области и задачи  
  * Источники знаний (откуда взяты правила)  
  * Формализация: какие НЕ-факторы, как представлены  
  * Примеры работы (ввод → вывод с объяснениями)  
  * Анализ валидации (если уровень 2+)  
  * (Уровень 3\) Результаты экспериментов, сравнение подходов, графики/таблицы  
* Для уровня 3: код интеграции (Python-скрипт, если есть)

Критерии оценки:

* Корректность формализации и вывода (40%)  
* Полнота БЗ (покрытие сценариев) (20%)  
* Объяснимость (качество трассировки) (15%)  
* Валидация и эксперименты (15%)  
* Документация (10%)  
* Бонусы: сравнительный анализ, интерактивный интерфейс

Защита: демонстрация работы системы на live-кейсе, объяснение выбора подхода, обоснование архитектуры.

# ЛАБОРАТОРНАЯ РАБОТА № 4

Реализация интерпретатора Prolog на Python

## **Краткое описание**

Разработать собственный интерпретатор подмножества языка Prolog на Python. Реализовать ключевые механизмы парадигмы логического программирования: парсинг, унификацию, SLD-резолюцию, backtracking. Продемонстрировать работу на примерах из ЛР1 (генеалогия).

## **Три уровня сложности**

Уровень 1 (базовый, 25 баллов):  
Минимальный рабочий интерпретатор:

* Парсинг:  
  * Факты: родитель(иван, мария).  
  * Простые правила: мать(X, Y) :- родитель(X, Y), женщина(X).  
  * Запросы: ?- мать(X, мария).  
  * Поддержка переменных (заглавные буквы), атомов (строчные), констант  
* Структуры данных:  
  * Представление термов (атом, переменная, составной терм)  
  * БЗ как список фактов и правил  
* Унификация:  
  * Алгоритм унификации двух термов с подстановками  
  * Проверка корректности (occur check опционален)  
* SLD-резолюция:  
  * Depth-first поиск с backtracking  
  * Выбор правила из БЗ, унификация, рекурсивное доказательство подцелей  
  * Вывод всех решений (через генератор или список)  
* Тестирование:  
  * Работает на примере генеалогии из ЛР1 (минимум 5 запросов)  
  * Корректно обрабатывает отсутствие решений

Уровень 2 (продвинутый, \+15 баллов):  
Расширение функциональности:

* Списки:  
  * Синтаксис: \[Голова|Хвост\], пустой список \[\]  
  * Унификация списков  
  * Встроенные предикаты: append/3, member/2, length/2  
* Арифметика:  
  * Предикат is/2: X is 2+3 → X=5  
  * Операторы: \+, \-, \*, /, mod  
  * Сравнения: \<, \>, \=\<, \>=, \=:=, \=\\=  
* Встроенные предикаты:  
  * true, fail  
  * \= (унификация как предикат)  
  * \\= (не унифицируется)  
* Оптимизация:  
  * Индексация первого аргумента для ускорения поиска правил  
  * Кеширование унификаций (мемоизация)  
* Отладка:  
  * Режим трассировки: вывод шагов резолюции  
  * Подсчёт числа унификаций, глубины рекурсии

Уровень 3 (экспертный, \+25 баллов):  
Продвинутые конструкции:

* Cut (отсечение):  
  * Оператор \! — отменяет backtracking  
  * Корректная реализация с учётом области действия  
* Negation as Failure:  
  * Оператор \\+ — отрицание через провал доказательства  
  * CWA (closed world assumption)  
* Динамические предикаты:  
  * assert/1, retract/1 — добавление/удаление фактов в runtime  
  * Поддержка изменяемой БЗ  
* Парсинг сложных термов:  
  * Вложенные структуры: точка(координата(X, Y))  
  * Операторы с приоритетами (опционально): X+Y\*Z  
* Сравнительное тестирование:  
  * Прогон тестов из ЛР1-3 на своём интерпретаторе VS SWI-Prolog  
  * Таблица: какие фичи работают, какие нет, разница в производительности  
* Документация кода:  
  * Архитектура системы (модули, классы)  
  * Комментарии к ключевым алгоритмам (унификация, резолюция)  
  * UML-диаграмма классов (опционально)

Состав материалов:

* Репозиторий на GitHub (или архив):  
  * prolog\_interpreter/ — исходный код (модули)  
  * tests/ — unit-тесты, примеры программ  
  * examples/ — примеры из ЛР1 (генеалогия) в формате интерпретатора  
  * README.md — описание, инструкция по запуску, примеры использования  
* Отчёт (PDF, 8-12 страниц):  
  * Архитектура системы (диаграммы, описание модулей)  
  * Алгоритмы: унификация (псевдокод/Python), резолюция (схема)  
  * Поддерживаемое подмножество Prolog (BNF-грамматика или описание)  
  * Примеры работы (скриншоты REPL с запросами)  
  * Тестирование: таблица результатов (какие тесты проходят)  
  * (Уровень 3\) Сравнение с SWI-Prolog, ограничения реализации  
  * Выводы: что сложно, что узнали, возможные улучшения  
* Видео-демонстрация (опционально, 3-5 минут): запуск интерпретатора, решение задачи из ЛР1

Требования к коду:

* Python 3.8+, стандартная библиотека (сторонние библиотеки — только для парсинга, если нужно)  
* Типизация (type hints) обязательна  
* Docstrings для классов и функций  
* Unit-тесты (pytest) для унификации и резолюции  
* Код должен проходить flake8/black (линтеры)

Критерии оценки:

* Корректность работы (50%):  
  * Унификация — 10%  
  * Резолюция \+ backtracking — 20%  
  * Встроенные предикаты (уровень 2+) — 10%  
  * Cut/NAF (уровень 3\) — 10%  
* Качество кода (архитектура, читаемость) (25%)  
* Тестирование (полнота тестов, покрытие) (15%)  
* Документация (README, отчёт) (10%)

Защита:

* Live-демонстрация: запустить интерпретатор, показать решение нетривиального запроса  
* Объяснение архитектуры: почему такие структуры данных, как работает backtracking  
* Code review: аспирант задаёт вопросы по коду (объяснить фрагменты)  
* Вопросы по теории: унификация, резолюция, отличия от императивных языков

# Общие требования к сдаче всех ЛР

Сроки:

* ЛР1: до конца 3-й недели  
* ЛР2: до конца 5-й недели  
* ЛР3: до конца 7-й недели  
* ЛР4: до конца 8-й недели (или зачётной недели)

Формат сдачи:

* Репозиторий GitHub (предпочтительно) или архив на облаке  
* Отчёт в PDF (обязательно)  
* Защита: очно или онлайн, запись на слоты через форму

Антиплагиат:

* Код проверяется через MOSS или вручную  
* Защита включает вопросы на понимание (если не можешь объяснить свой код — не засчитывается)

Пересдачи:

* Одна пересдача бесплатно (в течение недели после отклонения)  
* Вторая пересдача — с понижением балла на 20%
